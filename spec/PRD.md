# Stock Analysis Agent — PRD

> 开发前先写 Spec，把行为边界定清楚。这一步会让后续开发更稳定，避免陷入反复改提示词、反复补异常分支的状态。

---

## 1. 目标用户是谁

想看懂股票数据、希望获得分析参考但缺乏专业能力的个人投资者。

用户特征：
- 知道股票代码，但看不懂 K 线图和技术指标
- 想了解市场动态，但不知道从哪看起
- 需要有人帮他们"翻译"数据背后的含义
- 不需要自己去多个网站查数据，希望一站式获取分析结果

典型使用场景：
- 早上打开 Agent，看今天市场有什么重要消息（每日简报）
- 听说某只股票最近很火，想了解具体情况（全面分析）
- 手里有两只股票在犹豫，想对比一下（对比股票）

## 2. 高频任务有哪些

| # | 任务 | 用户会怎么说 | Agent 行为 |
|---|------|-------------|-----------|
| 1 | 查行情 | "茅台现在多少钱" | 调用 get_quote，返回价格和涨跌幅 |
| 2 | 看趋势 | "茅台最近一个月怎么样" | 调用 get_kline + get_news，K线和新闻按时间对齐 |
| 3 | 全面分析 | "帮我分析一下 AAPL" | 多工具组合：行情 + K线 + 估值 + 新闻，出综合报告 |
| 4 | 对比股票 | "茅台和五粮液哪个更值得看" | 两只股票的指标对比表 |
| 5 | 每日简报 | "今天市场有什么重要消息" | 并行查多只股票新闻，汇总市场简报 |

## 3. Agent 能调用哪些工具

从候选 7 个中选定 4 个（详见 [[tool-specs]]）：

| 工具 | 选中 | 说明 |
|------|------|------|
| 行情查询 | ✅ | 核心，所有任务都需要 |
| K 线分析 | ✅ | 趋势分析和全面分析需要 |
| 市值查询 | ❌ | 合并到行情查询的返回值中 |
| PE / PB 查询 | ✅ | 估值对比和全面分析需要 |
| 持仓分析 | ❌ | 需要用户输入持仓，留作扩展 |
| 新闻检索 | ✅ | 趋势分析、每日简报都需要 |
| 价格预警 | ❌ | 需要后台定时任务，留作扩展 |

工具通过 MCP 注册，SDK 自动发现，不需要手动告知模型工具列表。

## 4. 每个工具的参数和返回值
详见 [[tool-specs]]

选定的 4 个工具：
- **get_quote(symbol, market?)** → 实时价格、涨跌幅、成交量
- **get_kline(symbol, indicators, period?, interval?)** → 技术指标数据
- **get_valuation(symbol, metrics?)** → PE、PB、股息率等估值指标
- **get_news(symbol, keywords?, limit?, days?)** → 相关新闻列表

## 5. 哪些问题不能回答

| 拒答场景 | 原因 | Agent 应该怎么说 |
|----------|------|-----------------|
| "明天茅台会涨还是会跌" | 无法预测未来价格 | "我无法预测价格走势，但我可以帮你分析当前的技术面和基本面" |
| "帮我下单买 100 股" | 没有交易功能 | "我目前只提供数据分析，不支持交易操作" |
| "这只股票保证赚钱吗" | 不存在保证 | "投资有风险，我能提供数据参考，但无法保证收益" |
| "帮我内幕消息" | 违法且不存在 | "我只能基于公开信息进行分析" |
| "推荐一只必涨的股票" | 不存在必涨 | "我可以帮你对比多只股票的数据，但投资决策需要你自己判断" |

## 6. 哪些操作必须用户确认

本 Agent 是只读分析型，不执行任何写入操作，因此不需要用户确认。

如果未来扩展为交易型 Agent，以下操作必须确认：
- 任何涉及资金的操作（买入、卖出）
- 修改用户持仓信息
- 设置价格预警（需确认阈值合理性）

## 7. 工具失败时如何回复

| 失败类型 | 错误码 | 回复策略 |
|----------|--------|----------|
| 股票代码不存在 | SYMBOL_NOT_FOUND | 提示用户检查代码格式，举例说明正确格式 |
| 数据源不可用 | DATA_UNAVAILABLE | 告知用户当前无法获取，建议稍后重试 |
| 市场已休市 | MARKET_CLOSED | 返回最近可用数据，标注"数据截至 X 时间" |
| 网络超时 | TIMEOUT | 自动重试一次，仍失败则告知用户 |
| 数据点不足 | INSUFFICIENT_DATA | 用已有数据回答，标注数据范围限制 |
| 不支持的指标 | INVALID_INDICATOR | 告知用户支持的指标列表 |

错误处理策略（对应 Phase 3 学的 4 条恢复路径）：
1. 可重试错误（超时、数据源不可用）→ 自动重试一次
2. 参数错误（代码不存在、指标不支持）→ 提示用户修正
3. 数据不完整 → 用已有数据回答，标注限制
4. 不可恢复错误 → 告知用户当前无法完成，建议稍后重试

## 8. 如何判断回答质量

| 维度 | 标准 | 示例 |
|------|------|------|
| 数据准确性 | 引用的数据与数据源一致 | 股价、PE 值不能编造 |
| 推理逻辑 | 分析基于数据而非编造 | "RSI 38 接近超卖"✓ "我觉得会涨"✗ |
| 完整度 | 回答了用户的核心问题 | 用户问价格，不能只给代码 |
| 信息对齐 | 趋势分析中新闻和K线时间对齐 | 5月2日的新闻对应5月2日的K线变化 |
| 风险提示 | 涉及投资参考时有免责声明 | "以上分析仅供参考，不构成投资建议" |
| 格式清晰 | 输出结构化，易读 | 用表格对比，而不是一整段文字 |

## 设计原则
- 工具控制在 4 个，职责单一
- 工具过多会把问题分散到数据源、鉴权、异常处理和 UI 细节上，反而不利于验证 Agent 设计本身
- 第一个项目不建议做万能助手，范围太散也不容易验证效果
- 通过 MCP 注册工具，SDK 自动发现，不手动管理工具列表
- 只读分析，不做交易，降低风险
